可见，进一步调用`_runLoop`方法。`_runLoop`内部又根据当前是不是会话的首条消息分别调用了`agentLoop`和`agentLoopContinue`两个调用分支。不过`agentLoop`和`agentLoopContinue`最终都是调用`runLoop`来实现Agent Loop。
该函数实现了协调 LLM 调用、工具执行、steering 与 follow-up 的内外循环逻辑。主要的流程为：
* 1. 将 pending/steering 消息注入到 `currentContext`（如果存在）。
* 2. 调用 `streamAssistantResponse` 从 LLM 流式获取一条助手响应（包含部分更新事件）。
* 3. 如果响应包含 `toolCall`，调用 `executeToolCalls`，把工具结果追加到 `currentContext.messages`，并在同一请求内可能继续与 LLM 交互。
* 4. 重复上述步骤，直到没有更多工具调用、steering 或 follow-up 消息为止。

```typescript
// packages/agent/src/agent-loop.ts
async function runLoop(...){
	//外循环，处理当前提问的循环，如果又新的插队进来就先处理后再结束
	while (true) {
			// 内循环，如果还有工具要执行或者存在插队信息就不断执行LLM问答-工具执行步骤
			while (hasMoreToolCalls || pendingMessages.length > 0) {
				...
				//与大模型进行一轮交互
				const message = await streamAssistantResponse(currentContext, config, signal, stream, streamFn);
newMessages.push(message);
        // 从大模型返回结果解析工具调用清单
				const toolCalls = message.content.filter((c) => c.type === "toolCall");
				hasMoreToolCalls = toolCalls.length > 0;
				const toolResults: ToolResultMessage[] = [];
				if (hasMoreToolCalls) {
				// 逐个工具执行并得到执行结果
					const toolExecution = await executeToolCalls(currentContext.tools, //当前可用的所有工具
							message, // 这里包括本turn的工具调用列表
							signal,
							stream,
							config.getSteeringMessages,
							);

					toolResults.push(...toolExecution.toolResults);
				  steeringAfterTools = toolExecution.steeringMessages ?? null;
					for (const result of toolResults) {
						currentContext.messages.push(result);
						newMessages.push(result);
					}
					
				}

				...
			}
	}
}
```

`streamAssistantResponse`是`pi-mono`封装的大模型问答函数，他主要的作用是统一了各大主流大模型的输入和输出格式。返回的`message`是`AssistantMessage`对象。其主要字段为：
- `role`: "assistant"
 - `content`: Array<TextContent | ThinkingContent | ToolCall>
	 - TextContent: { type: "text", text: string }
	- ThinkingContent: { type: "thinking", thinking: string, redacted?: boolean }
	- ToolCall: { type: "toolCall", id: string, name: string, arguments: Record<string, any> }
	 - `api` / `provider` / `model`: 来源信息
- `usage`: 使用统计（tokens / cost 等
 - `stopReason`: "stop" | "length" | "toolUse" | "error" | "aborted"
- `timestamp`: 时间戳（毫秒）


上面的代码的从返回的`content`中筛选出类型为'ToolCall'的信息即为大模型返回的工具调用。
+ **抽取工具调用清单**：`const toolCalls = message.content.filter((c) => c.type === "toolCall");`
+ **判断是否需要工具调用**：`hasMoreToolCalls = toolCalls.length > 0;`

工具的执行则是使用封装的`executeToolCalls`函数，它对照已注册的工具列表，检查诊断每一个待执行工具的格式，然后异步执行工具调用，返回每个工具调用的结果。
